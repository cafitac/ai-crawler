# Risks and Constraints

## Technical Risks

### 1. Some sites cannot be replayed with HTTP only

Reason:

- JS-generated tokens
- signed requests
- proof-of-work challenges
- encrypted payloads
- browser fingerprint checks
- WebAssembly signing logic

Mitigation:

- classify as `browser_required`
- support browser fallback for discovery/limited execution
- allow user-provided cookies/session
- document confidence score

### 2. Anti-bot systems change frequently

Risk:

- recipes break
- proxy/IP reputation changes
- TLS/header fingerprints become stale

Mitigation:

- recipe test command
- failure classifier
- repair loop
- fingerprint/backend abstraction

### 3. AI hallucination

Risk:

- AI invents endpoints or fields
- AI produces invalid recipe
- AI ignores constraints

Mitigation:

- structured output schema
- recipe validation
- dry-run every generated recipe
- evidence-based prompts
- no direct execution of arbitrary AI code

### 4. Prompt injection from scraped pages

Risk:

- webpage content tries to instruct AI

Mitigation:

- sanitize hidden/instruction-like content
- treat page content as data, not instructions
- send compact evidence bundles instead of raw pages

### 5. High crawl speed can cause operational harm

Risk:

- overload target site
- trigger bans
- legal/compliance issues

Mitigation:

- rate limiter
- robots.txt option
- allowed domain guard
- default conservative concurrency
- clear legal disclaimer

### 6. Browser probe is still heavy

Risk:

- Playwright dependency is large
- CI/container setup complexity
- memory usage

Mitigation:

- make browser extras optional
- keep HTTP execution independent
- support HAR import so probe can run elsewhere

### 7. Network capture may expose secrets

Risk:

- cookies, bearer tokens, CSRF tokens, proxy credentials, or private API responses may be captured.

Mitigation:

- classify captured headers/cookies/storage as secrets
- redact before logs/snapshots/failure reports
- store sensitive runtime state separately from recipe files
- add tests for redaction

### 8. External LLM/provider dependency can make workflows brittle

Risk:

- model output changes
- provider outage
- cost spikes
- rate limits

Mitigation:

- keep deterministic fallback where possible
- use MockLLMClient in tests
- enforce structured output
- cache evidence and generated recipe drafts
- expose manual CLI flow

### 9. Recipe drift over time

Risk:

- target API schema/pagination/token flow changes after recipe creation.

Mitigation:

- `test-recipe` command
- scheduled recipe health checks
- failure classifier
- repair loop
- versioned recipes

### 10. Over-scoping into a generic scraping framework

Risk:

- project becomes a clone of Scrapy/Scrapling/Playwright wrappers.

Mitigation:

- keep focus on AI-driven network API discovery and recipe compilation
- browser execution remains fallback, not primary product
- avoid building broad parser/browser features unless needed by recipe generation

---

## Legal/Ethical Constraints

This project must be framed as a tool for authorized data access, research, testing, and automation.

Guidelines:

- respect robots.txt when configured
- respect terms of service
- do not bypass access controls without permission
- do not collect personal data without lawful basis
- do not provide features framed as credential theft, evasion of security systems, or unauthorized access

Preferred wording:

- network-first crawling
- request replay for authorized workflows
- browser probe for API discovery
- compliance-aware crawling

Avoid product messaging like:

- hack any site
- bypass all protections
- undetectable scraping

---

## Product Risk

Scrapling already exists and has MCP scraping support.

Differentiation must remain clear:

- Scrapling = scraping framework + browser/stealth tools + MCP
- ai-crawler = AI-powered network crawler compiler

If ai-crawler becomes just another MCP fetch server, it loses differentiation.

---

## MVP Scope Control

Do not implement in MVP:

- CAPTCHA solving
- stealth browser bypass
- distributed cluster
- login automation
- proxy marketplace integrations
- GUI dashboard
- arbitrary JS execution generated by AI

MVP should prove one thing:

Browser probe can discover an API, and curl-cffi can crawl it faster without the browser.
